# Hallucination Analysis in RAG and GraphRAG Systems

## 📌 Project Overview
Large Language Models (LLMs) augmented with retrieval mechanisms (RAG) often generate hallucinations—statements not supported by the retrieved information . 
This semester project aims to study, in a concrete and empirical way, how the use of structured knowledge graphs in **GraphRAG** systems affects the frequency and nature of hallucinations compared to a classical vector-based **RAG** .

**Institution:** EURECOM
**Supervisors:** Carmelle Meli Songuon, Thibault Ehrhart, Pasquale Lisena, Raphael Troncy.

## 🎯 Objectives & Tasks
The project is structured around three main tasks:
1. **Task 1 - Dataset Expansion:** Generating a balanced set of simple and multi-hop questions using 50% text-based generation and 50% graph-based generation based on the *OSKGC* and *Text2KGBench-Lettria* datasets.
2. **Task 2 - Pipelines Setup:** Implementing two question-answering pipelines (Standard Vector RAG and GraphRAG) using LangChain.
3. **Task 3 - Evaluation:** Running controlled experiments to evaluate hallucinations quantitatively (using the Mirage framework ) and qualitatively (analyzing failure cases such as unsupported facts, incorrect relations, or invented entities ).
4. *(Bonus)* **Grounding Mechanisms:** Implementing graduated grounding mechanisms to explicitly control the degree of creativity during generation and study the trade-offs.


## 🛠️ Datasets
*   [Text2KGBench-LettrIA](https://drive.google.com/file/d/1GCEEHzHkTkaMbjgrKfYgIcZVE4-VixPe/view?usp=drive_link): A curated benchmark for ontology-guided Text-to-Knowledge-Graph systems containing 4,860 sentences and over 14,000 triples.
*   [OSKGC](https://github.com/HeraclesWang/OSKGC): A benchmark designed for knowledge graph construction based on a predefined ontology schema containing 10,183 text entries.


## 📁 Dataset File Structure Explained

The datasets (like Text2KGBench-LettrIA) are divided into specific thematic categories (e.g., `airport/`, `artist/`). Inside each category folder, the data is strictly compartmentalized to separate training, testing, and business logic. Here is what each file represents:

### 📄 JSON Lines (`.jsonl`)
*Unlike standard JSON, each line in a `.jsonl` file is an independent JSON object, allowing for efficient processing of large datasets without overloading the RAM.*

* **`train.jsonl`**: The standard training dataset containing pairs of raw text and their corresponding factual triples.
* **`train_bis.jsonl`**: The "Extended" training dataset. It contains the base training data supplemented with synthetic data augmentation (generated by LLMs like Gemini 2.5 Pro) to improve model robustness and scale the training volume.
* **`test.jsonl`**: The testing dataset containing unseen text data. This is the input we feed to our RAG and GraphRAG pipelines to evaluate their generalization capabilities.
* **`ground_truth.jsonl`**: The absolute reference (Golden Labels). It contains the perfect expected triples for the `test.jsonl` set. Our evaluation framework (MIRAGE) compares our pipeline's output against this file to detect hallucinations and calculate structural accuracy.

### 🕸️ Turtle (`.ttl`)
*A standard Semantic Web format used to describe RDF graphs.*

* **`ontology.ttl`**: The schema and business rules for the category. It does not contain raw text. Instead, it defines the strict hierarchy and allowed relationships (e.g., specifying that a `President` is a subclass of `Politician`, or defining the domain and range of a specific relation). This is the structural backbone that prevents the GraphRAG from hallucinating impossible relationships.
